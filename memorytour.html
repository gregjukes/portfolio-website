Memory Tour

Overview

Context: Memory Tour is a student project created for Career Foundry’s UX Immersion course.

Design objective: Enable players to enjoy social scavenger hunts for real objects and locations in their own cities

Role: UX Designer

Research

Talking to users

Surveys and interviews gave me perspective on how folks think about scavenger hunts and their relationships with social games in person and online.

Organizing responses from my interviews into behaviors/attitudes, needs/goals, frustrations, quotes, and facts structured my data so I could create user personas for Memory Tour based on real-world insights.

Working with personas

Journey Maps outline scenarios that help answer the question: “Why will a user like Esmeralda want to create an event with Memory Tour?”

Translating users’ need to pages of the application

Interviews along and user personas and flows identified three primary uses of the app:

  1. Creating scavenger hunts for friends

  2. Playing scavenger hunts

  3. Sharing memories from a group activity

Data from a card sort using Optimal Workshop allowed me to refine the language and categories that structure the site based on the three primary uses:

  1. Create is where users build scavenger hunts

  2. Tours are the games themselves

  3. Profile contains user-specific content including memories, tours created/in progress, and tours played 

Ideation

Wireframes

I use an assortment of tools to create wireframes and prototypes at different levels of fidelity so I can cleanly and efficiently communicate my evolving design concepts.

Using Balsamiq to turn my onboarding sketches into mid-fidelity wireframes helped me create a cleaner image of these pages incorporating standard UI elements.

For my initial prototype of the tour creation sequence, I went directly from rough, low-fidelity sketches to high-fidelity designs in Adobe XD that I could implement right away in user testing.

Making space for change

My initial low- and mid-fidelity wireframes of the tour search page helped me clarify content and actions, but the design was too cluttered. I quickly sketched out a few ideas for how my location search might look based on existing map apps and search pages (Google Maps, Yelp, Facebook), trying out a few different organization patterns so I could confidently build high-fidelity wireframes in Adobe XD.

Testing

Remote, moderated usability test

Test Objectives and Goals

  • The goal of this test is to asses the basic functionality of the prototype and its learnability for new users interacting with the app for the first time.

  • Determine if participants can intuitively navigate through the app.

  • Observe how users attempt to look for a tour to play, create a tour of their own, and share photo memories from their homepage. Can they accomplish these tasks?

  • Gauge users’ initial satisfaction with the product.

Participants and Tasks

Six participants took part in the usability test. Most were in our target demographic of 18-35 years old living in an urban area, though there were a couple older participants and some folks lived in suburban or rural areas. At this stage of testing, these variations simply provide a wider range of responses rather than creating concern about missing key insights from focusing only on the target audience.

This test asked participants to complete three of the primary tasks in the app as identified by our user personas:

  1. Find a tour

  2. Create a tour

  3. Share memories with friends

Results

Familiar is good, creating is hard

  • Testers found the prototype most satisfying and intuitive to use on pages where the design aligns closely with existing products: Tours is modeled after common map apps and Yelp, Share Memories is modeled after iOS photo sharing.

  • This satisfaction is reflected in their comments and ratings for the standard ease question asked after each task. Tasks 1 and 3 (find a tour and share a memory) had high SEQ averages and comments including: “it did exactly what I thought it would do” and “it was how all sharing in the world happens”

  • Testers had more trouble working through the steps of creating a tour and some found the process frustrating: “I spent most of [the task] confused”

  • Creating a Tour is a key activity for the app and something that will make it unique. The issues uncovered by the test participants offer important insights into what changes need to be made.

Revisions

Guiding users through creating a tour

The overarching issue with the initial design (top) was that users felt unsure about what to do, so the new design (bottom) integrates more descriptive text throughout each screen to guide users in making a tour. By making the language more specific and detailed, I hope to help users past the learning curve of their first experience. Several testers indicated that the creation was fun and made sense once they’d figured it out. Hopefully these redesigns will allow them to figure it out with more certainty on first glance.

Key changes include:

  • Breaking the first create page into two parts: one page for naming the tour followed by a building page

  • Changing wording on key buttons to reduce confusion

  • More descriptions on each page, button, and input field

Next Steps

Memory Tour’s next steps for iteration include:

  1. Testing the current prototype to see if the changes made to the create sequence have made it easier for users experiencing the app for the first time to make a tour and if any new issues have been created.

  2. Turning low-fi wireframes of the actual gameplay sequence for playing a tour into a mid- or hi-fi prototype to test this aspect of the app with users.

  3. Reviewing the style guide and design language system to make sure they accurately describe the current look and feel of the app.

  4. Consulting the design brief and business requirements document to look for any fundamental business, product, or design concepts that have not yet been addressed.
